# 商业 UI/UX

## 核心原则

商业界面工作先关注用户任务是否清晰，再关注交互是否安全，最后才是视觉表达。一个看起来精致、但不能帮助用户完成正确任务的界面，不是好设计。视觉风格服务于业务链路；除非用户明确要求视觉探索，否则它不是主要交付物。

## 必需工作流

1. 判断任务模式和输入契约：领域、受众、平台、核心链路、品牌身份、技术栈、已有设计系统和约束；简报不完整时记录假设和未解决缺口。
2. 判断视觉来源模式：客户指定、参考图、既有设计系统，或在没有风格来源时使用项目推断。
3. 修改文件前先检查项目上下文。除非用户明确要求重做，否则保留现有框架、组件库、路由、状态、命名和视觉约定。
4. 按 UX -> UI -> GUI 顺序工作：先定任务链路与状态，再定层级和响应式布局，最后定可操作感和控件状态。
5. 只在用户请求的界面范围内实现。避免无关重构、新框架和不服务任务的装饰复杂度。
6. 使用项目可用检查验证结果。视觉工作优先做桌面和移动视口的浏览器或截图验证。

## 输出契约

设计类任务输出 UX 结构、必要的角色与路径模型、视觉来源和依据、UI 规格、风险说明、假设或未解决缺口与验收检查。

实现类任务输出修改过的文件、已执行验证和本地无法验证的缺口。

评审类任务先列问题：严重程度、可用时的文件和行号、用户影响、建议修复方式。

## 默认商业规则

- 优先遵循已有设计系统。
- 修复既有产品界面时，保留信息架构、组件语义和视觉 token；只修复失败状态或失败链路，不重绘无关界面。
- 使用 8px 间距节奏，卡片圆角默认不超过 8px，除非已有系统不同。
- 文本对比度满足 WCAG AA，优先使用清晰、熟悉的控件而不是自创交互。
- 每个交互元素在适用时都需要 default、hover、active、focus、disabled、loading 和 error 处理。
- 受保护或破坏性操作需要硬门槛：disabled 或 aria-disabled 状态、确认输入、权限检查、审批等待或撤销路径。
- 除非用户要求落地页，否则优先构建真实可用的产品界面，而不是营销首页。
- 不要用模糊装饰图像隐藏关键产品信息。
- 面向高密度或高风险流程时，要让审计上下文、字段校验、移动端任务长度和可衡量的质量提升保持可见。

## 红旗信号

发现以下情况时，停下来重新限定范围：

- 用户只说“做漂亮一点”，但没有用户任务、受众或产品上下文。
- 界面用颜色或动画替代信息层级。
- 页面缺少 loading、error、empty、disabled 或破坏性操作处理。
- 生成页面在现有产品中引入了无理由的新设计语言。
- 密集流程只是因为提示词列全了状态才显得完整，但移动端流程、审计上下文或字段校验仍然薄弱。
- 界面依赖欺骗性设计模式、虚假紧迫感、欺骗性默认值或不可访问交互。
