# 质量门槛与验收清单

## 1. 完成前最低门槛

在声称 UI/UX/GUI 工作完成前，必须至少检查：

- 核心用户任务是否明确。
- 当任务涉及多角色或不完整简报时，角色责任、主路径、异常/恢复路径和未解决缺口是否可追溯。
- 视觉来源是否明确：客户指定、参考图、设计系统或项目推断。
- 若使用项目推断模式，是否说明配色、按钮、表单、表格、导航和不采用风格的依据。
- 视觉策略是否服务任务链路，而不是替代 UX 结构、状态覆盖和风险处理。
- 修复既有界面时，是否保留信息架构、组件语义、视觉 token、路由和状态模型。
- 主行动和次行动是否清晰。
- 布局是否符合当前产品或项目约定。
- 关键文本是否可读且不溢出。
- 交互控件是否有必要状态。
- 加载、空、错误和权限状态是否覆盖。
- 破坏性操作是否有保护。
- 移动和桌面视口是否都可用。
- 可访问性基础是否达标。
- 已执行可用的本地验证命令。

## 2. UX 检查

| 检查项 | 通过标准 |
| --- | --- |
| 用户目标 | 页面能一句话说明服务什么用户任务 |
| 角色与路径 | 多角色或协作型任务能区分责任、主路径和异常/恢复路径 |
| 主路径 | 用户能在 1-3 个主要步骤中完成核心任务 |
| 导航上下文 | 用户知道自己在哪里、能去哪里 |
| 空状态 | 空数据时给出原因和下一步 |
| 错误恢复 | 错误提示说明发生了什么和如何修复 |
| 权限状态 | 权限不足时不表现为系统故障 |
| 破坏性操作 | 删除、重置、覆盖等有确认或撤销 |
| 修复边界 | 既有产品修复只修补失败链路、缺失状态、字段校验、可访问性或响应式问题，不重绘无关界面 |
| 高风险审计 | 审计信息、决策依据、操作者和状态变化靠近相关操作 |
| 字段级校验 | 高风险表单的必填、权限、理由和确认字段有明确校验与恢复说明 |
| 假设缺口 | 简报不完整时，关键假设和需要产品确认的缺口能在界面或交付说明中追溯 |

## 3. UI 检查

| 检查项 | 通过标准 |
| --- | --- |
| 视觉策略 | 视觉来源清楚，样式选择与行业、用户、平台和任务匹配 |
| 视觉定位 | 视觉风格服务于业务任务，不把配色、渐变、装饰图或字体变化当成主要交付成果 |
| 视觉层级 | 标题、内容、操作、辅助信息权重明确 |
| 间距 | 主要 spacing 遵循 8px rhythm |
| 色彩 | 颜色有语义，不只为了装饰 |
| 对比度 | 正文和关键控件满足 WCAG AA |
| 字阶 | 面板内文字不过大，辅助文字不过小 |
| 密度 | 信息密度符合目标用户，不盲目留白或堆叠 |
| 一致性 | 与已有产品或设计系统一致；修复既有界面时保留信息架构、组件语义和视觉 token |
| 可观察改善 | 改动能明确减少状态、审计、校验、响应式或可访问性遗漏 |

## 4. GUI 交互检查

| 控件 | 必查状态 |
| --- | --- |
| Button | default、hover、active、focus、disabled、loading |
| Input | default、focus、filled、disabled、error、helper text |
| Select/Menu | closed、open、selected、focus、empty、disabled |
| Table/List | loading、empty、row hover、selected、bulk action、error |
| Modal/Dialog | focus trap、close、confirm、cancel、escape behavior |
| Toast/Alert | severity、dismiss、timeout、screen-reader behavior |

## 5. 响应式检查

至少检查：

- 375px 移动宽度。
- 768px 平板宽度。
- 1280px 桌面宽度。

通过标准：

- 无横向溢出，除非表格明确采用横向滚动。
- 文本不遮挡按钮、图标或相邻内容。
- 主行动在小屏仍可找到。
- 导航、筛选和批量操作在小屏有合理降级。
- 移动端长流程有步骤摘要、分段锚点、粘性操作或详情页降级，不只是桌面布局纵向堆叠；固定或粘性操作不得覆盖正文、状态说明或字段错误。
- 表格横向溢出必须是有意设计，并保留行身份、关键状态和主要操作。

## 6. 可访问性检查

- 所有图标按钮有 `aria-label` 或可见文本。
- 表单控件有 label。
- 错误文本与字段关联。
- 焦点样式可见。
- Tab 顺序符合阅读和操作顺序。
- 不只靠颜色表达状态。
- 图片有适当替代文本，装饰图片应隐藏给辅助技术。

## 7. 工程检查

根据项目情况运行可用命令：

```bash
npm run lint
npm run typecheck
npm run test
npm run build
```

如果项目没有这些命令，应说明实际执行了什么检查。视觉任务建议补充浏览器截图或手动视口检查。

普通 UI/UX/GUI 工作的完成检查到这里为止。公开安装包不包含维护脚本、展示页或独立复测产物；请使用用户项目本身可用的 lint、typecheck、test、build、浏览器检查或截图检查来验证实际交付。
