# 使用说明文档

## 1. 这个 skill 解决什么问题

当 Codex 使用者需要设计、审查或实现商业 UI/UX/GUI 时，本 skill 帮助 Codex 从“生成一个好看的页面”转向“生成一个可用、可信、可维护、可验证的界面”。

适合的请求包括：

- “帮我做一个 SaaS 后台首页。”
- “把这个表单页面优化得更商业化。”
- “检查这个页面的 UI/UX 问题。”
- “为这个 React 项目新增一个订单管理页面。”
- “设计一个移动端审批流。”
- “修复这个既有后台页面的状态、字段校验和移动端问题，但不要重做设计语言。”

## 2. 普通 UI 工作读取路径

普通设计、评审或实现任务只需要从 `SKILL.md` 进入，并按当前任务读取必要文档：

- 设计或实现 UI：读取 `docs/01-commercial-ui-ux-rules.md`，完成前读取 `docs/06-quality-gates.md`。
- 需要判断边界、拒绝或重新限定范围：读取 `docs/02-skill-boundaries.md`。
- 需要解决美感、可用性、商业目标和无障碍冲突：读取 `docs/03-design-constitution.md`。
- 没有品牌规范、参考图或明确风格：读取 `docs/07-visual-strategy-inference.md`。
- 只有行业、垂直领域或宽泛产品方向：按需读取 `docs/09-industry-taxonomy.md` 和 `docs/10-industry-template-library.md`。

维护、复测、截图和发布准备材料不属于公开安装包，也不作为普通 UI 任务的默认上下文。

## 3. 推荐输入格式

用户可以用自然语言，也可以提供下面的结构化输入：

```markdown
项目类型：
任务类型：新建 / 修复 / 评审
目标用户：
交付平台：
核心任务：
技术栈：
已有设计系统：
保留范围：
品牌调性：
视觉来源：
必须包含的页面或组件：
不希望出现的风格：
```

示例：

```markdown
项目类型：B2B SaaS 客户管理后台
任务类型：新增页面
目标用户：销售主管和客户成功团队
交付平台：Web 响应式，桌面优先
核心任务：查看客户健康度、筛选高风险客户、创建跟进任务
技术栈：React + Tailwind，已有 lucide-react
已有设计系统：浅色、8px 圆角、蓝色为主行动色
保留范围：沿用现有导航、表格密度、按钮样式和筛选交互
品牌调性：可信、清爽、高效
视觉来源：未提供，按项目内容自动推断
必须包含：客户列表、风险标签、筛选器、批量分配
不希望出现：营销式 hero、大面积渐变、过度卡片化
```

## 4. Codex 的默认处理方式

当信息足够时，Codex 应直接推进，并在输出中标注假设。当信息不足且会导致完全不同的结果时，Codex 应先问一个关键问题。

默认假设：

- 平台：Web 响应式。
- 风格：克制、现代、商业可信。
- 视觉来源：没有品牌规范、参考图或已有系统时，自动启用项目推断模式。
- 既有界面修复：默认保留信息架构、组件语义、视觉 token、路由和状态模型，只修复失败链路、缺失状态、字段校验、可访问性或响应式问题。
- 用户：中等技术熟悉度。
- 布局：任务型应用界面，而不是营销落地页。
- 技术：遵循当前仓库。

## 5. 视觉来源模式

用户可以选择四种方式之一：

- 客户指定：提供品牌色、Logo、字体、组件库或设计稿。
- 参考图：上传参考图、竞品截图或风格图。
- 设计系统：要求遵循已有 Ant Design、Material UI、shadcn/ui、Tailwind token 或企业组件库。
- 项目推断：不提供视觉资料，让 Codex 根据项目内容自动生成视觉策略；视觉策略服务于任务链路，不替代 UX 结构、状态覆盖和风险处理。

如果用户没有说明，Codex 默认使用项目推断模式，并在输出中说明推断依据。

## 9. 好请求与差请求

好请求：

```markdown
帮我为一个面向财务分析师的 Web 数据看板设计信息架构和 React 页面，要求高密度、可筛选、支持加载和空状态。
```

差请求：

```markdown
做得高级一点。
```

差请求不是不能处理，但 Codex 应主动补充业务假设，并避免只堆砌视觉风格词。
