# 商业 UI/UX/GUI 规则文档

## 1. 设计目标

商业界面的首要目标是帮助目标用户在真实业务场景中安全、高效、低认知负担地完成任务。视觉表现服务于任务完成、信任建立、信息理解和业务转化。

## 2. 输入契约

开始设计或实现前，尽量从用户请求、项目文件和上下文中提取以下信息。缺失信息不会自动阻塞，但高风险缺失必须说明假设。

| 维度 | 必要信息 | 默认假设 |
| --- | --- | --- |
| 行业领域 | SaaS、金融、医疗、电商、教育、工具、内容等 | 通用 SaaS / 工作台 |
| 目标用户 | 专业用户、大众用户、运营、管理者、开发者等 | 中等技术熟悉度的办公用户 |
| 交付平台 | Web、移动端、桌面端、小程序、响应式 | Web 响应式 |
| 核心任务 | 用户要完成的 1-3 个关键动作 | 单页核心任务优先 |
| 品牌资产 | Logo、主色、语气、字体、现有设计系统 | 克制、现代、可信 |
| 技术栈 | React、Vue、Next.js、Tailwind、组件库等 | 遵循当前仓库 |
| 数据状态 | 真实数据结构、空状态、加载、错误、权限 | 需要补齐所有关键状态 |

## 3. 视觉策略规则

当用户没有提供品牌规范、参考图或明确风格时，必须启用项目推断模式。不要因为缺少参考图就停止设计，也不要随意套用流行视觉效果。

Codex 应根据行业、目标用户、平台、核心任务、风险等级、数据密度和既有组件库，自动生成视觉策略。视觉策略至少说明：

- 视觉来源：客户指定、参考图、设计系统或项目推断。
- 推断依据：为什么该行业、用户和任务适合这种视觉方向。
- 配色：主色、辅助色、背景、surface、边框和语义色。
- 字体与字阶：标题、正文、辅助文本、数字和表格。
- 控件样式：按钮、表单、表格、卡片、导航、弹窗。
- 交互状态：hover、focus、active、disabled、loading、error、selected。
- 不采用的风格：例如营销 hero、大面积渐变、玻璃拟态、低对比度灰字。

详细规则见 `docs/07-visual-strategy-inference.md`。

## 4. UX 规则

### 4.1 任务优先

每个页面必须回答三个问题：

- 用户当前在哪里。
- 用户现在能做什么。
- 用户完成主要任务的下一步是什么。

如果一个视觉元素不能帮助用户理解、决策或行动，应删除或降级。

### 4.2 主次动作清晰

每个主要工作区只能有一个视觉最强的主行动。次行动、取消、返回、导出、筛选、更多操作必须在视觉权重上低于主行动。

破坏性操作必须有防错机制，例如二次确认、撤销、软删除、危险色、明确后果描述或权限限制。

### 4.3 状态完整

面向用户的流程必须覆盖：

- 初始状态。
- 加载状态。
- 空状态。
- 成功状态。
- 错误状态。
- 部分失败状态。
- 权限不足状态。
- 离线或超时状态，当产品场景需要时。

### 4.4 信息架构

页面结构应从业务优先级而不是视觉装饰出发。常见顺序为：

1. 全局导航或上下文标题。
2. 当前页面目标和关键状态。
3. 主要操作。
4. 核心内容或数据。
5. 次级工具，如筛选、排序、导出、批量操作。
6. 帮助、说明或低频设置。

修复既有产品界面时，默认保留原有信息架构、组件语义、视觉 token、路由和状态模型。除非用户明确要求重做，否则只修补失败链路、缺失状态、可访问性、字段校验和风险提示，不把修复任务变成整页重绘或营销化改版。

### 4.5 高密度工作台

高密度工作台不能只靠把所有组件塞进一屏来显得专业。金融风控、运维、审计、医疗随访、客服质检等场景，应优先保证用户能稳定完成判断、比较、记录和追责。

- 审计轨迹、决策依据、操作者、时间、状态变化和未解决风险必须靠近相关操作，不应被挤到不易发现的侧栏角落。
- 高风险字段校验必须显性化：字段缺失、格式错误、权限不足、审批人缺失、理由不足和后果确认都应有字段级提示。
- 受保护操作必须有硬状态：说明文字不能替代 `disabled`、`aria-disabled`、确认输入、权限门槛或审批等待态。
- 移动端长流程要有分段、锚点、步骤摘要、粘性主操作或详情页降级，不能只把桌面三栏纵向堆到底；粘性或固定操作不得遮挡正文、状态说明或表单错误。
- 表格在小屏采用横向滚动时，必须保留行身份、关键状态和主操作；否则应转卡片、摘要列表或详情页。
- 可观察增益必须能被验证：新增规则应让状态、校验、审计、响应式或无障碍问题更少，而不是只让文案更像设计说明。

## 5. UI 规则

### 5.1 布局与间距

- 默认使用 8px spacing rhythm：4、8、16、24、32、40、48、64。
- 4px 仅用于紧凑控件、图标与文本的光学校正。
- 数据密集型后台可以更紧凑，但必须保持可读性和点击目标。
- 页面 section 不应被过度包装成浮动卡片。卡片用于重复项、表单组、弹窗、工具面板和确实需要边界的内容。
- 不要把卡片嵌套在卡片里，除非已有设计系统明确这样做。

### 5.2 视觉层级

视觉层级必须通过尺寸、字重、间距、位置、色彩和分组共同建立。不要只靠颜色表达重要性。

页面标题、区块标题、表格列名、辅助文本和错误文本必须有稳定字阶。紧凑面板里不要使用英雄级大字。

### 5.3 色彩

- 正文与背景对比度至少满足 WCAG AA：普通文本 4.5:1，大文本 3:1。
- 品牌色优先用于主行动、选中态、关键状态和少量高亮。
- 成功、警告、错误、信息色必须具备语义一致性。
- 禁止用低对比度浅灰承载关键文本。
- 避免单一色相统治整个界面，尤其是没有业务理由的紫蓝渐变、米色、深蓝和棕橙主题。

### 5.4 字体

- 优先使用项目已有字体系统。
- 没有既有系统时，Web 默认使用现代系统字体栈。
- 不随视口宽度线性缩放字体。
- 字距保持正常，避免负字距。
- 长文本必须可换行，不得溢出按钮、卡片或表格单元格。

## 6. GUI 交互规则

每个交互控件必须根据适用性定义以下状态：

| 状态 | 要求 |
| --- | --- |
| Default | 可识别用途和默认层级 |
| Hover | 鼠标悬停时有轻微反馈 |
| Active / Pressed | 点击瞬间有明确反馈 |
| Focus | 键盘访问时有可见焦点 |
| Disabled | 说明不可用原因或通过上下文表达 |
| Loading | 防止重复提交，保留布局稳定 |
| Error | 与具体字段或操作关联，说明修复方式 |
| Selected | 当前选择清晰可见，不能只靠颜色 |

图标必须使用用户熟悉的隐喻。可用图标时优先用图标按钮，并提供 tooltip 或 aria-label。不要用自造符号替代通用操作。

## 7. 响应式规则

- 桌面端优先保证扫描、比较和批量操作效率。
- 移动端优先保证触控目标、单手路径和纵向信息流。
- 表格在移动端需要明确策略：转卡片、横向滚动、列裁剪或进入详情页。
- 移动端长流程必须控制认知负担：保留当前位置、任务进度、关键摘要和可恢复路径。
- 关键操作不能只存在于 hover。
- 弹窗在移动端应考虑底部抽屉、全屏页或分步流程。

## 8. 无障碍规则

- 使用语义化 HTML 或平台原生语义。
- 所有表单控件需要可感知标签。
- 错误提示要与字段关联，并说明如何修复。
- 焦点顺序必须符合视觉和任务顺序。
- 键盘用户必须能完成主要任务。
- 图标按钮必须有可访问名称。
- 动画不能阻碍使用，必要时支持 reduced motion。
- 不要滥用 ARIA；优先使用原生语义。

## 9. 代码生成规则

- 遵循当前仓库技术栈和组件库。
- 不为单个页面引入新的 UI 框架。
- 设计 token 应集中定义，不在组件中散落硬编码。
- 组件应拆分在真实复用点，不为抽象而抽象。
- 交互逻辑、数据状态和视觉状态应保持一致。
- 示例数据必须明显是示例，不得伪装成真实用户或真实业务结论。
