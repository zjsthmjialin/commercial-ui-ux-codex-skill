# Skill 边界文档

## 1. 适用范围

本 skill 适用于以下任务：

- 设计商业软件界面，包括 SaaS、CRM、ERP、BI、金融后台、运营平台、移动业务应用和工具型产品。
- 为已有前端项目新增或改进页面、组件、表单、仪表盘、导航、设置页和工作流。
- 修复既有产品界面的失败链路、缺失状态、字段校验、响应式、可访问性和高风险操作问题，同时保留原有信息架构、组件语义和视觉 token。
- 审查 UI/UX/GUI 方案，指出可用性、可访问性、视觉层级、响应式和状态覆盖问题。
- 生成设计规则、设计 token、组件规范、交互说明、验收清单和实现建议。
- 将粗糙的“好看一点”需求转成可执行的界面需求。

## 2. 不适用范围

本 skill 不应主导以下任务：

- 纯艺术创作、海报、插画、品牌视觉探索和非功能性视觉实验。
- 只要求独立配色、字体、Logo、装饰图或换肤方案，且没有具体产品界面、用户任务或交互状态需要处理。
- 游戏玩法、美术资产、角色设计和沉浸式 3D 场景，除非任务重点是产品界面。
- 法律、医疗、金融等领域的专业合规判断。可以设计界面，但不能宣称合规结论。
- 欺骗用户、隐藏成本、诱导误操作、制造虚假稀缺或阻碍取消订阅的界面。
- 未经授权模仿具体品牌、产品或受版权保护的视觉系统。

## 3. 需要先询问的情况

默认应主动推进，但遇到以下情况要先问清楚：

- 用户要求重做已有产品的整体视觉语言，但仓库中已有设计系统或品牌规范。
- 用户只要求“换个风格”“改配色”或“做高级一点”，但没有说明要服务的用户任务、失败链路或保留范围。
- 任务涉及高风险领域，并且界面文案可能影响医疗、法律、投资或安全决策。
- 用户要求生成真实品牌、真实人物或真实企业的仿冒界面。
- 目标平台、技术栈或交付格式会决定完全不同的实现方式，而且上下文无法推断。
- 用户需求同时包含多个大型子系统，如“做完整 CRM、移动端、官网、管理后台和设计系统”。

## 4. 可以合理默认的情况

以下缺失信息可以先用默认值推进，并在输出中标注假设：

- 未说明行业：默认通用 SaaS 或工具型产品。
- 未说明平台：默认 Web 响应式。
- 未说明视觉风格：默认克制、现代、可信、信息清晰。
- 未说明组件库：默认遵循当前仓库；若无仓库上下文，使用原生语义结构和轻量样式。
- 未说明用户类型：默认中等技术熟悉度的办公用户。
- 未提供品牌规范或参考图：启用项目推断模式，根据行业、用户、平台和任务自动生成视觉策略。

## 5. 视觉来源边界

- 客户指定内容优先于自动推断。
- 用户上传参考图时，只提取风格特征，不复制受保护设计。
- 项目已有设计系统时，不用项目推断模式覆盖已有组件库。
- 项目推断模式必须说明依据和不采用的风格，不能只输出“高级、现代、科技感”等抽象词。
- 用户不满意自动推断结果时，应允许用户改用客户指定模式或参考图模式重新生成。

## 6. 禁止的设计行为

- 使用欺骗性设计模式提高转化，例如隐藏取消、误导确认、默认勾选付费项。
- 用低对比度、微小文字或复杂路径隐藏重要信息。
- 让危险操作没有二次确认、撤销或明确后果。
- 让关键功能只依赖颜色、hover、动画或图片表达。
- 将装饰性视觉置于任务完成之前。
- 生成看似真实的用户数据、财务数字或健康结论而不标注示例。

## 7. 工程边界

- 修改已有项目时，只改与请求相关的文件。
- 不擅自替换框架、组件库、状态管理和构建工具。
- 不把一次性页面需求过度抽象为大型设计系统。
- 不删除用户已有改动。
- 不在没有验证的情况下声称“生产就绪”。

## 8. 视觉边界

- 不用纯装饰渐变、光斑、漂浮球体填补设计空洞。
- 不让视觉主题被单一色相控制，除非品牌规范要求。
- 不使用难以检查真实产品的模糊、暗色、裁切过度图片。
- 不用大面积营销式 hero 代替用户请求的实际应用界面。
